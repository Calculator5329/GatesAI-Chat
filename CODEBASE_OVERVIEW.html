Architecture Overview

GatesAI Chat is a Vite, React, TypeScript single-page app wrapped by Tauri. A Go bridge process is expected beside the app for workspace file and process operations. The client keeps app state in MobX stores and calls service modules for persistence, providers, tools, bridge transport, search, and image generation.

Chat send path

EditorialComposer
  -> ChatStore.sendMessage(...)
     -> append user message and derive selected model/context/tool set
     -> ProviderStore + ModelRegistry choose provider/model config
     -> services/llm/router creates LlmProvider
     -> provider streams normalized LlmChunk events
     -> ChatStore updates assistant message, activity rows, token usage, tool calls
     -> tools execute through services/tools with ToolContext facades
     -> BridgeStore/BridgeClient handle workspace operations when needed
     -> persistence queues localStorage and optional workspace snapshot writes

Conventions

AreaConventionSources
Error handlingPreserve graceful fallbacks. Parse unknown JSON into narrow shapes before stores consume it. Do not replace diagnostic `console.warn`/`console.error` with a logger until a logger module is approved.ERROR_HANDLING_NOTES.md, `src/services/*` parser helpers.
State managementStores own mutations and observable state. Components should derive view state and avoid broad store observation where a narrower prop or computed value exists.RENDER_PERF_NOTES.md, `src/stores/*` headers.
Dynamic models`ModelRegistry` is the owner for provider dynamic catalogs. Provider-specific stores expose getters/setters that sync through the registry.STATE_OWNERSHIP_NOTES.md, `ModelRegistry`, `OllamaStore`, `OpenRouterStore`.
Bundle hygieneHeavy UI sections and tool/provider modules can be loaded with dynamic imports when behavior stays the same. Analyzer mode is available through `npm run analyze`.IMPORT_BUNDLE_CLEANUP_NOTES.md, `vite.config.ts`.
NamingUse domain-specific names for private UI helper types. Do not rename public exports or persisted fields without approval.NAMING.md, NAMING_CLEANUP_NOTES.md.
File organizationComponents stay under feature folders; service modules own IO and parsing; stores own state transitions.Observed in `src/components`, `src/services`, and `src/stores`.
