GatesAI Chat — how the codebase fits together

A desktop-first LLM chat app: React 19 + MobX SPA wrapped in Tauri, talking to a bundled Go bridge for filesystem and tool execution, plus optional Ollama and ComfyUI sidecars. This page maps the layers, the data flow, and what the in-progress trim plan is removing.

React 19 MobX 6 Tauri 2 Vite 8 Vitest OpenRouter ✓ keep Ollama ✓ keep ComfyUI ✓ keep A1111 / cloud images — dropping

Overview

The product is a single chat surface plus a settings menu. Messages stream from a model provider; the model can call tools that touch a real workspace on disk, run code, manage notes, or queue local image generation. Everything lives in a Tauri window, but the heavy lifting — file IO, terminal, git, python, sqlite — happens in a separate Go bridge process the app spawns.

Persistence pattern

Each persistable store has a MobX autorun that calls a per-key save* helper in src/services/*Storage. There's an emergency compaction path for the chat snapshot that trims tool result bodies to ~600 chars (edge-preserving) when localStorage throws a quota error.

// Pattern repeated across stores
makeAutoObservable(this)
autorun(() => saveSnapshot(this.snapshot))   // observes .snapshot getter

Service layer

Each provider implements one interface and emits a streaming async iterable.

interface LlmProvider {
  ready(): boolean
  stream(req: LlmRequest, signal: AbortSignal): AsyncIterable<LlmChunk>
}

type LlmChunk =
  | { kind: 'text'; text: string }
  | { kind: 'tool_call'; call: ToolCall }
  | { kind: 'done' }

router.ts picks the provider for a model id. Direct keys preferred, OpenRouter as fallback. The trim plan reduces the routable surface to openrouter, ollama, and the synthetic local-image provider that bypasses the LLM and goes straight to ComfyUI.

wireFormat.ts flattens internal messages (assistant rows carry their own toolResults) into the wire shape providers expect (separate tool role rows). contextCompaction.ts reduces oversized tool results when usage crosses ~90% of context window for opted-in models.

The registry exposes a single contract; ChatStore feeds the ctx at execute time, holding references to bridge, notes, summary, imageGen stores, and the active thread id.

interface Tool {
  def: ToolDef                                  // JSON-Schema for the model
  execute(args, ctx): Promise<string | ToolExecuteResult>
  meta?: { category, resultPolicy, isReadOnly }
}

Tool selection per turn is heuristic: memory, thread, and artifact always; bridge-y tools when the user text mentions files/code/etc.; image tools when a vision/draw cue is present. Falls open (all tools) if nothing matches — biased toward capability.

Today: 5 backends — local-comfy, local-a1111, cloud-openrouter, cloud-openai, cloud-gemini. Trim plan: keep only local-comfy. The dispatcher in imageBackend.ts will collapse to one branch; ImageJobStore, ComfyClient, and the workflow override path stay intact.

BridgeClient wraps the WS as a typed RPC: request(op, data, onEvent?) with optional streaming-event callback for things like terminal output. Disconnected calls reject with BridgeOfflineError; tools translate to friendly user messages. Attachments upload via multipart to /upload and the chat references them as workspace paths.

chatLog.ts emits per-thread events (turn.start, tool_call, tool_result, turn.end). Routed through the bridge to workspace/logs/<threadId>.log when online; queues locally when offline.

Chat turn flow

What happens when the user hits send. The loop is the heart of ChatStore.runTurn().

sequenceDiagram
  autonumber
  participant U as User
  participant C as Composer
  participant CS as ChatStore
  participant R as LlmRouter
  participant P as Provider
  participant T as ToolRegistry
  participant B as Bridge
  U->>C: types message
  C->>CS: sendMessage(text, attachments)
  CS->>CS: append UserMessage, new AbortController
  loop until no tool_calls (or MAX rounds)
    CS->>R: resolve(modelId)
    R-->>CS: { provider, providerModelId }
    CS->>CS: composeSystemPrompt + flattenForWire
    CS->>P: stream(request, signal)
    P-->>CS: text chunks → StreamingTextBuffer
    P-->>CS: tool_call (buffered)
    alt has tool_calls
      CS->>T: execute each (parallel)
      T->>B: fs / terminal / git / …
      B-->>T: result
      T-->>CS: toolResults[]
    else no tool_calls
      CS->>CS: final reply, break
    end
  end
  CS->>CS: autorun → saveSnapshot()

Step by step

  1. Send. EditorialComposer calls chat.sendMessage(text, attachments). A UserMessage is appended; an AbortController is keyed by thread id so users can cancel without affecting other threads.
  2. Resolve provider. router.resolve(modelId) returns { provider, providerModelId } or throws NoProviderConfiguredError (UI shows the "add an API key" banner).
  3. Compose context. profile.composeSystemPrompt({ runtimeContext, threadContext, recentSummaries }); messages flatten to wire format and image attachments resolve to base64 via the bridge.
  4. Pick tools. toolRegistry.toolDefsForTurn({ userText, bridgeOnline, imageGenAvailable }) returns the JSON-Schema set the model gets to see this round.
  5. Stream. StreamingTextBuffer batches text chunks (~48 chars or 16ms) before mutating message.content — keeps React from re-rendering per character.
  6. Collect tool calls. When the stream ends with tool_call chunks, they're stored on the assistant message; tools execute in parallel via Promise.all; results land on the same message as toolResults[].
  7. Loop. Same assistant message gets another round with the new tool results in scope, until the model produces a final reply.
  8. Persist. Mutation triggers the autorun in ChatStore; saveSnapshot() writes gatesai.state.v1; on quota error the emergency compactor kicks in.
Why one assistant message per turn (not many)?

Multi-round tool loops used to produce N assistant messages plus M role: 'tool' messages. Persistence migration collapses those into a single assistant message that owns its toolCalls and toolResults arrays. Renderer + wire-format conversion treat that as the source of truth.

The trim plan — what's keep / drop

Side-by-side view of the foundation the other agent is shaping. Keep is what stays manually testable; drop is removed (not hidden) so the surface stays honest.

Keep ✓

  • Chat providers: OpenRouter, Ollama
  • Local images: ComfyUI (Draft / Normal / Upscale)
  • Direct-image models in the model picker
  • Assistant context: custom system prompt, memory, notes, thread mgmt, summaries
  • Workspace tools: workspace, fs, inspect_file, terminal, python_inline, sqlite_query, query_script, git, artifacts
  • Surfaces: threads, model picker, OpenRouter API key, Local (Ollama/Comfy), Workspace, Gallery, Appearance

Drop ✗

  • Direct cloud LLM SDKs: Anthropic / OpenAI / Gemini / Groq direct adapters
  • Local OpenAI-compatible chat backend
  • Cloud image backends: CloudImageCard, openAi/openRouter/gemini image clients + tests
  • A1111 backend + progress adapter + UI
  • Prompt-enhancement controls (unless wired this pass — recommend remove)
  • Agent menu's fake/non-persisted Model defaults, Voice & tone, planned-tool rows
  • RoutingCard (collapse to nothing useful with one cloud provider)
  • Stale plan/spec language about cloud images / fal / A1111

Risks to watch during the trim

Saved threads with removed model ids

Older threads carry modelId values pointing at direct-provider entries. The migration must fall back to DEFAULT_MODEL_ID so the renderer/composer don't blow up.

Image backend value churn

imageGenStorage needs to coerce cloud-* and local-a1111 to local-comfy on load. Any unhandled value will break the Local menu badge.

Vision / describe_image

With direct OpenAI/Anthropic gone, vision relies on Ollama or OpenRouter. The fallback chain in describe_image needs review so it doesn't crash when neither is configured.

Tests of trimmed surface

tests/services/{llmRouter,modelCatalog,router}.test.ts and tests/components/menu/GatesMenu.test.ts will all need updates; they're already shown as modified — good signal.

Bridge contract is in scope

The trim does not touch the bridge protocol or the workspace tool stack. Don't accidentally edit shared types in src/services/bridge/*.

Doc drift

docs/tech_spec.md and docs/roadmap.md still describe the larger surface. Mark removed pieces historical, not current.

Onboarding playbook (read in this order)

  1. src/app/App.tsx — see how RootStore mounts and the menu/chat split works.
  2. src/stores/RootStore.ts — the wiring diagram in code form. Note the lazy provider injections.
  3. src/stores/ChatStore.ts — read sendMessage and runTurn; everything else is supporting cast.
  4. src/services/llm/router.ts — small file, makes the routing rules concrete.
  5. src/services/llm/openrouter.ts + ollama.ts — pick one provider impl to grok the streaming contract.
  6. src/services/tools/registry.ts + one tool (e.g. fs.ts, terminal.ts) — see the def/execute shape.
  7. src/services/bridge/client.ts — RPC wrapper over the WS.
  8. src/components/editorial/EditorialChat.tsx + EditorialMessage.tsx — render side; they're MobX observers reading store state.
  9. src/services/image/imageBackend.ts + src/stores/ImageJobStore.ts — the image path top-to-bottom.
  10. tests/stores/ChatStore.test.ts + tests/stores/toolLoop.test.ts — these double as executable docs for the loop.
Run order for verification (matches the trim plan): focused tests per layer (npx vitest tests/services/llm etc.) → full npm run ci → npm run tauri:dev for the manual smoke list (OpenRouter chat, Ollama chat, system prompt, memory, workspace tools, ComfyUI direct image, image_generate, Gallery/Lightbox).
