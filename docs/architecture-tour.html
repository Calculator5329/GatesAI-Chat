GatesAI Chat is a React 19 + MobX desktop chat app wrapped in Tauri. The current foundation routes model traffic through OpenRouter, Ollama, or a synthetic local-image provider, and sends privileged workspace operations through a separate bridge sidecar when it is available.

Services

The service layer is where the app leaves pure state and does work: HTTP streaming, local storage, bridge RPC, tool handlers, image queue adapters, and diagnostics.

LLM

router.ts resolves models to providers. openrouter.ts and ollama.ts implement the streaming contract; localImage.ts exists as the synthetic provider id for direct image models.

Bridge

client.ts wraps the bridge WebSocket as typed request/response RPC. Attachments and image bytes are resolved through bridge helpers.

Tools

Each tool exports a JSON-schema definition and an execute() function. The registry chooses a per-turn subset from user text and runtime availability.

Image

imageBackend.ts currently has one branch: local-comfy. comfyClient.ts queues workflows and reads outputs.

Local runtime

localRuntimeService.ts calls Tauri commands; autoDetect.ts finds likely install paths; storage keeps local runtime settings.

Persistence

Storage helpers load, sanitize, migrate, and save JSON snapshots. The stores decide when to persist through MobX autoruns.

Tool Registry

The model never gets arbitrary app internals. It receives tool definitions from the registry, then ChatStore executes requested calls through a constrained context object.

Critical Flows

These are the routes to trace when debugging most behavior: one normal chat turn and one local image job.

Chat turn

Composer sends

EditorialComposer calls chat.sendMessage(text, attachments).

ChatStore appends

A user message is added, streaming state is prepared, and an AbortController is stored per thread.

Router resolves

LlmRouter.resolve(modelId) returns OpenRouter, Ollama, or local-image if the provider is ready.

Prompt is composed

System prompt, recent summaries, runtime context, messages, attachments, and tools become the provider request.

Provider streams

Text chunks feed StreamingTextBuffer; tool-call chunks are collected.

Tools execute

If calls exist, registry execution writes results back onto the same assistant message and the loop continues.

Snapshot saves

MobX autorun writes gatesai.state.v1, compacting huge tool results if localStorage rejects the full snapshot.

Image job

Entry

Either the user selects a direct image model or the model calls image_generate.

Queue

ImageJobStore.enqueue() creates a pending job with prompt, dimensions, count, seed, and Comfy mode.

Runner

The serial runner promotes one job to active and builds a backend snapshot from ImageGenStore.

ComfyUI

dispatchImageGenerate() resolves local-comfy and sends the workflow through ComfyClient.

Artifact write

The final image bytes are written to /workspace/artifacts/ through bridge fs.write.

History

Done, failed, and cancelled jobs move into persisted history for Gallery and chat cards.

Context Engine

The app does more than stream messages. Each provider round rebuilds a fresh context envelope: runtime facts, user memory, recent summaries, artifact README instructions, tool schemas, attachments, and context-budget compaction.

Runtime context

buildRuntimeContext() injects local time, timezone, ISO timestamp, bridge online/offline state, platform/version when known, workspace path semantics, and terminal cwd rules.

src/services/chat/runtimeContext.ts

Artifact README context

When the bridge is online, ChatStore scans /workspace/artifacts/**/README.md, sorts paths, skips unreadable files, and caps both per-file and total injected text.

src/services/chat/artifactReadmeContext.ts

Memory layers

The system prompt includes durable profile facts, recent summaries from other threads, and per-thread context. There is no embeddings/RAG layer today.

src/stores/UserProfileStore.ts

Wire expansion

Storage keeps one assistant message per user turn; flattenForWire() expands it back into assistant/tool rows for providers.

src/services/llm/wireFormat.ts

Budget pressure

Before each request, the store estimates system prompt + wire messages + tool schemas + reserved output. At about 90% of the model window, it compacts large tool results.

src/services/llm/contextCompaction.ts

Tool failure log

Failures are detected centrally after execution and logged with call id, tool name, result preview, redacted args, read-only classification, bridge state, and duration.

src/services/chat/toolFailureLog.ts
Stored thread
one assistant row per user turn
Prompt envelope
profile, summaries, runtime, artifact READMEs
Wire request
flattened messages + selected tool schemas
Budget guard

A useful debugging rule: if the model seems confused about files, time, workspace paths, or old tool output, inspect the request-building path in ChatStore.buildTurnRequest(), buildRuntimeContext(), loadArtifactReadmeInstructions(), and flattenForWire() before changing UI code.

Common Extension Recipes

1

Add a tool

Create src/services/tools/name.ts, export a Tool, use only ToolContext facades, add one toolRegistry.register() line, then test through tests/services/tools* or a focused store test.

2

Add local runtime state

Keep process work in Tauri/service wrappers, expose only a small store facade, persist under the runtime storage helper, and avoid mirroring URLs into unrelated stores.

3

Add a chat feature

Start in ChatStore only if the feature changes turn orchestration. Otherwise add a service helper, store state, then render it through observer components.

4

Add persisted prefs

Define a tiny storage helper, normalize unknown values on load, use a store autorun to save, and add migration behavior for old values rather than trusting localStorage.

5

Add provider behavior

Implement the LlmProvider streaming contract, translate vendor chunks into text, tool_call, and done, and keep provider-specific quirks out of ChatStore.

6

Add image behavior

Preserve the queue/artifact contract: image work should enqueue jobs, persist final files into /workspace/artifacts, and let ImageJobCard observe progress by job id.
