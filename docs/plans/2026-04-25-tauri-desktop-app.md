# Tauri Desktop App Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Ship GatesAI Chat + bridge as a single double-clickable desktop app (Windows `.exe` first, macOS/Linux later) so non-developers can install it without Node, Go, or a terminal.

**Architecture:** Tauri shell wraps the existing React/Vite UI in a native window using the OS's webview. The existing Go bridge ships unchanged as a Tauri "sidecar" — a separate binary the Rust shell spawns on launch and kills on exit. The chat still talks to the bridge over `127.0.0.1:7331` HTTP+WebSocket, so `BridgeStore` and `BridgeClient` need no changes.

**Tech Stack:** Tauri 2.x, Rust (only for the ~50-line shell), existing Vite/React UI, existing Go bridge as sidecar binary, NSIS installer for Windows.

**Repos involved:**
- `C:\Users\et2bo\Desktop\Projects\GatesAI Chat` — chat UI, will host Tauri scaffolding under `src-tauri/`
- `C:\Users\et2bo\Desktop\Projects\gatesai-bridge` — Go bridge, gets a cross-compile script for sidecar binaries

**Key constraints:**
- Bridge binary name must follow Tauri's convention: `gatesai-bridge-<target-triple>.exe` (e.g. `gatesai-bridge-x86_64-pc-windows-msvc.exe`).
- Bridge default port is 7331; if it's already running (someone left the old `.cmd` launcher up), the app must detect and reuse instead of failing on bind.
- Bridge config lives at `~/.gatesai/bridge.json` and is auto-created — no installer step needed for first-run state.
- Tauri 2.x requires explicit capability/permission declarations to spawn sidecars; v1 syntax does not apply.

**Out of scope for v1:** macOS/Linux installers (plan mentions cross-compile but Windows is the only verified target), code signing, auto-update server, custom workspace-picker UI in the shell.

### Task 3: Add bridge cross-compile script (in bridge repo)

**Files:**
- Create: `C:/Users/et2bo/Desktop/Projects/gatesai-bridge/scripts/build-sidecar.ps1`
- Modify: `C:/Users/et2bo/Desktop/Projects/gatesai-bridge/README.md` (add a "Building for Tauri sidecar" section)

**Why a script:** Tauri's `externalBin` mechanism requires the binary filename to include the Rust target triple. We script this so the chat repo can call it cleanly.

**Step 1: Write the build script**

`scripts/build-sidecar.ps1`:
```powershell
# Builds gatesai-bridge with the filename Tauri expects for sidecar bundling.
# Usage:   pwsh scripts/build-sidecar.ps1 [-OutDir <path>]
# Default OutDir is ../GatesAI Chat/src-tauri/binaries

param(
  [string]$OutDir = (Join-Path $PSScriptRoot "..\..\GatesAI Chat\src-tauri\binaries")
)

$ErrorActionPreference = "Stop"
$repoRoot = Resolve-Path (Join-Path $PSScriptRoot "..")
Set-Location $repoRoot

# Resolve the host's Rust target triple. We invoke `rustc -vV` so the
# script works even if the user has multiple toolchains installed.
$rustcOut = & rustc -vV
$triple = ($rustcOut | Select-String '^host:').ToString().Split(' ')[-1].Trim()
if (-not $triple) { throw "could not determine Rust host triple from rustc -vV" }

# Sanity: GOOS/GOARCH from the triple.
switch -Wildcard ($triple) {
  "*windows*"      { $env:GOOS = "windows"; $ext = ".exe" }
  "*darwin*"       { $env:GOOS = "darwin";  $ext = "" }
  "*linux*"        { $env:GOOS = "linux";   $ext = "" }
  default          { throw "unsupported triple: $triple" }
}
switch -Wildcard ($triple) {
  "x86_64-*"  { $env:GOARCH = "amd64" }
  "aarch64-*" { $env:GOARCH = "arm64" }
  default     { throw "unsupported arch in triple: $triple" }
}

New-Item -ItemType Directory -Force -Path $OutDir | Out-Null
$dest = Join-Path $OutDir ("gatesai-bridge-" + $triple + $ext)

Write-Host "Building bridge for $triple → $dest"
& go build -trimpath -ldflags "-s -w" -o $dest ./cmd/gatesai-bridge
if ($LASTEXITCODE -ne 0) { throw "go build failed" }

Write-Host "OK: $dest"
```

### Task 5: Spawn the bridge sidecar on app start

**Files:**
- Modify: `src-tauri/src/lib.rs`

**Design:** On app startup (`setup` hook), check if something is already listening on `127.0.0.1:7331`. If yes → reuse it (do nothing). If no → spawn the sidecar with `--listen 127.0.0.1:7331` and store the child handle in app state so we can kill it on exit.

**Step 1: Add `reqwest` (blocking, tiny) for the health check, and a Mutex for the child handle**

In `src-tauri/Cargo.toml`:
```toml
reqwest = { version = "0.12", default-features = false, features = ["blocking", "rustls-tls"] }
```

**Step 2: Implement the spawn-or-reuse logic in `lib.rs`**

```rust
use std::sync::Mutex;
use tauri::Manager;
use tauri_plugin_shell::process::CommandChild;
use tauri_plugin_shell::ShellExt;

struct BridgeChild(Mutex<Option<CommandChild>>);

fn bridge_already_running() -> bool {
    let client = reqwest::blocking::Client::builder()
        .timeout(std::time::Duration::from_millis(500))
        .build();
    let Ok(client) = client else { return false };
    matches!(
        client.get("http://127.0.0.1:7331/health").send(),
        Ok(r) if r.status().is_success()
    )
}

#[cfg_attr(mobile, tauri::mobile_entry_point)]
pub fn run() {
    tauri::Builder::default()
        .plugin(tauri_plugin_shell::init())
        .manage(BridgeChild(Mutex::new(None)))
        .setup(|app| {
            if bridge_already_running() {
                eprintln!("[gatesai] bridge already running on 7331; reusing");
                return Ok(());
            }
            let sidecar = app.shell()
                .sidecar("gatesai-bridge")?
                .args(["--listen", "127.0.0.1:7331"]);
            let (_rx, child) = sidecar.spawn()?;
            app.state::<BridgeChild>().0.lock().unwrap().replace(child);
            eprintln!("[gatesai] spawned bridge sidecar");
            Ok(())
        })
        .run(tauri::generate_context!())
        .expect("error while running tauri application");
}
```

### Task 6: Kill the sidecar on app exit

**Files:**
- Modify: `src-tauri/src/lib.rs`

**Why a separate task:** if we leave orphan bridge processes after window close, the user gets stuck the next time they launch (ports busy, file locks, etc.). Tauri does not kill spawned children automatically.

**Step 1: Wire a window-close handler that takes the child and kills it**

In `lib.rs`, extend the builder before `.run(...)`:
```rust
.on_window_event(|window, event| {
    if let tauri::WindowEvent::Destroyed = event {
        let app = window.app_handle();
        if let Some(child) = app.state::<BridgeChild>().0.lock().unwrap().take() {
            let _ = child.kill();
            eprintln!("[gatesai] bridge sidecar killed");
        }
    }
})
```

**Step 2: Manual verification**

1. `npm run tauri:dev`, wait for bridge online indicator
2. Close the app via the X button
3. In a separate terminal: `Get-Process gatesai-bridge -ErrorAction SilentlyContinue` — expected: nothing
4. Re-launch immediately — bridge should come up cleanly

If a bridge process lingers, check whether the close handler fires (add a `dbg!()`). Sometimes you need `tauri::WindowEvent::CloseRequested` instead of `Destroyed` depending on platform behavior — adjust if so.

**Step 3: Commit**

```bash
git add src-tauri/src/lib.rs
git commit -m "fix: kill bridge sidecar when window closes"
```

### Task 9 (stretch): macOS / Linux builds

Out of scope for v1 ship. Notes for later:
- Cross-compiling Go to mac/linux is one-line (`GOOS=darwin GOARCH=arm64 go build ...`); the script in Task 3 already detects host triple, just needs invocation per target.
- Tauri supports `dmg` and `appimage`/`deb` bundle targets; add to `tauri.conf.json > bundle.targets` per platform.
- macOS unsigned `.app` bundles trigger Gatekeeper warnings. Code signing requires an Apple Developer account ($99/yr). Acceptable to ship unsigned for friends-and-family v1 with a "right-click → Open" workaround note in README.
- Tauri builds must run on the host OS for that target — no cross-OS bundling. CI later (GitHub Actions matrix) is the clean answer.

Skip until someone actually asks for non-Windows.
