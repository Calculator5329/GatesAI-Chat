# GatesAI semantic memory implementation plans

Generated by the `improve` skill on 2026-07-19 against commit `dee51c2`.
Execute in the order below. Each executor must read its plan fully, honor its
STOP conditions, and update the status row when it finishes.

These plans cover only the first item in the narrowed roadmap: semantic memory
/ RAG. Search, the model-compat auto-runner, and the future database/library
layer remain separate roadmap work.

## Outcome

GatesAI should remember across all retained conversations without silently
treating old text as truth. Recall must be measurable, complete across the hot
and archived chat tiers, fast enough not to interrupt local chat, inspectable in
the conversation, and controllable by source and item. The implementation stays
local: Ollama creates embeddings and browser IndexedDB stores the derived index.

## Dependency notes

- 001 comes first because the current tests prove mechanics, not recall quality.
  The later plans must compare against a frozen, publication-safe corpus rather
  than tune thresholds by feel.
- 002 precedes ranking work because the current source snapshot contains empty
  stubs for archived threads. Improving ranking before fixing corpus completeness
  would optimize only the hot tier and then regress when the source repository
  changes.
- 003 precedes the polished UI because the UI must disclose stable structured
  retrieval records, not parse the current formatted text output.
- 004 owns the product surface, persisted provenance, exclusions, and safe prompt
  boundary. It consumes the structured index and retrieval contracts established
  by 002–003.

## Product defaults carried into execution

- Keep the routing floor: Ollama embeddings only. Do not add cloud embedding
  providers or a generic custom endpoint.
- Saved memory facts remain the small, explicit, always-on “About the user”
  layer. Semantic recall is a separate evidence layer over facts, notes, and
  retained conversations.
- Automatic recall may remain on by default once an embedding model is ready,
  but it must be visible, reversible, source-scoped, and conservative on
  no-match queries.
- Conversation excerpts are historical data, not instructions and not verified
  facts. Preserve author role and source provenance end to end.
- Desktop is the product. Web Lite must remain stable and honest, but no new
  browser-only semantic-memory experience is required.
- Do not index workspace files in this epic. File/document search belongs with
  the later search or database/library work.

## Findings considered and rejected

- **Restore the pre-trim Semantic Memory component unchanged:** rejected. It
  exposed a status, model textbox, auto-inject checkbox, and rebuild button but
  did not solve recall quality, provenance, archived threads, or per-source
  control.
- **Add cloud embeddings as a fallback:** rejected. It violates the documented
  offline-capable product direction and creates an avoidable privacy boundary.
- **Add LLM query rewriting, HyDE, or a cross-encoder immediately:** deferred.
  They add latency, model routing, and failure modes before a local hybrid
  baseline exists. The evaluation harness may justify one later, but 003 must
  prefer the simplest pipeline that clears the gate.
- **Adopt an external vector database now:** rejected. The corpus is local and
  already IndexedDB-backed; first prove that a worker-based hybrid scan misses
  the latency budget before adding a dependency or sidecar.
- **Index tool outputs and attachments by default:** rejected. They are large,
  often transient, and carry a wider prompt-injection/data-leak surface. A later
  source type can add them only with an explicit contract and benchmark cases.
